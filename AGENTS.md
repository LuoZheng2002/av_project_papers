AGENTS.md

This document provides build/test/lint commands and code-style conventions for agentic coding agents that operate on this repository.

1) Build / lint / test commands

- Install dependencies (node toolchain used by repo tools):
  - npm install
  - (or) bun install if using Bun: bun install

- Run linting (if present):
  - npm run lint
  - If using ESLint directly: npx eslint . --ext .js,.ts

- Run static type checks (TypeScript):
  - npx tsc --noEmit

- Run the whole test suite:
  - npm test
  - or: npx jest --runInBand

- Run a single test file (recommended for iterative debugging):
  - npx jest path/to/testFile.test.ts -i
  - or (mocha): npx mocha path/to/testFile.test.js

- Run a single test by name/regex (Jest):
  - npx jest -t "test name substring" -i

- Run a single test case in Vitest:
  - npx vitest run path/to/testFile.spec.ts -t "case name"

- Run Type-aware tests (ts-node + mocha / ts-jest):
  - npx ts-node/register path/to/test

- Run formatting:
  - npm run format
  - or: npx prettier --write .

Notes:
- Use the package.json scripts where available (npm run test, npm run lint) so local conventions and environment variables are respected.
- Use the -i or --runInBand flags when running single tests in CI-like (single-threaded) environments to avoid flaky parallel behavior.

2) Repo structure notes (quick)

- .opencode/: supporting tools and SDKs (node packages). Treat generated files as third-party; avoid editing node_modules directly.
- paperwork/: documentation and generated summaries — markdown files maintained by agents.
- Keep code and docs changes separate in commits when possible.

3) Code style guidelines

General principles
- Prioritize clarity and maintainability. Make the minimal local change that achieves the goal and prefer small, well-tested diffs.
- Favor explicitness over cleverness. Comments should explain why (not what) when code is non-obvious.

Formatting
- Use Prettier for JavaScript/TypeScript formatting. Settings: 2-space indent, single quotes, trailing comma as "es5", print width 100.
- Run: npx prettier --write "**/*.{js,ts,md,json,css,mdx}" before committing.

Imports and module boundaries
- Use absolute imports only if the project is configured for them (tsconfig paths). Prefer relative imports for local modules (./, ../).
- Order imports: builtin node modules, external packages (alphabetical), internal modules (alphabetical). Separate groups with a blank line.
- Avoid importing large modules at top-level in files that are loaded in CLI paths where startup time matters—defer require/import inside functions where appropriate.

Types (TypeScript)
- Prefer explicit types on public function signatures and exported APIs. Use type inference for small local variables when it improves readability.
- Use readonly and const where applicable. Prefer immutable data structures where possible.
- Keep interfaces small and focused. Use type aliases for unions and discriminated unions for variant types.

Naming conventions
- Files: kebab-case for markdown and documentation; snake_case or kebab-case for per-paper summaries (keep existing repo conventions).
- Variables and functions: camelCase. Classes and types: PascalCase. Constants: UPPER_SNAKE_CASE only for environment or build-time constants.
- Test files: suffix with .spec.ts or .test.ts and place next to the implementation where practical.

Error handling and logging
- Fail early with clear error messages. Validate inputs at boundaries and throw informative errors (include contextual data where safe).
- Use try/catch at API boundaries and convert low-level errors to higher-level errors with context. Do not swallow errors silently.
- For libraries or utilities, return Result-style objects or throw exceptions consistently—document the choice in the module README.

Testing
- Keep tests deterministic and fast. Avoid network calls and heavy filesystem operations in unit tests; use fixtures and mocks.
- Use seeded randomness for reproducible tests if randomness is required.
- Aim for clear Arrange/Act/Assert structure. Keep tests focused: one behavior per test.

Documentation & comments
- Update relevant docs (paperwork/ or README) for any public-facing change. When changing MD outputs, keep autogenerated vs authored sections clearly separated.
- Use docstrings for exported functions (JSDoc/TSDoc) describing inputs, outputs, throw conditions, and side effects.

Commits and PRs
- Keep commits atomic and focused. Each commit should have a single purpose.
- Use descriptive commit messages: short (50 chars) summary line, optional body with motivation and non-obvious details.

Code review expectations
- Include a short summary of the change and any assumptions in the PR body.
- Add unit tests for behavioral changes and run lint/format before requesting review.

Security and secrets
- Never commit secrets or credentials. Use environment variables and .env files excluded by .gitignore for local testing only.

Agent-specific operational rules
- Agents must not modify files outside their assigned scope without explicit instruction. For example: when asked to update paperwork/ai_generated/per_paper_summary, only touch files in that directory and coordinating index files as requested.
- Agents must use the provided command-line tools (npm scripts) rather than inventing custom build steps when an official script exists.
- When creating or editing markdown summaries, preserve existing formatting conventions (heading levels, metadata blocks) and make minimal changes to unrelated sections.

Cursor / Copilot rules
- No project Cursor rules (.cursor/rules/) or Copilot instructions (.github/copilot-instructions.md) were detected in the repository root. If such files are added, agents must read them and follow them before further changes.

Appendix — quick checklist for an agent before committing
- Run lint and format: npm run lint && npm run format (or corresponding commands).
- Run the affected tests (single test or suite): npx jest path/to/changed.test.ts -i
- Ensure no secrets added and update paperwork/ or docs if public API changed.
- Keep changes atomic and include a clear commit message describing the "why".
